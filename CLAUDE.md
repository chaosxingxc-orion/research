## Engineering Rules

### Rule 1 — Think Before Acting
Surface assumptions, name confusion, and state tradeoffs before writing a single line of code, prose, or analysis. If multiple valid interpretations exist, present them — never pick one silently. If the requirement is unclear, stop and ask.

**Strongest valid reading.** When a request is ambiguous, default to the strongest reading: "review" → comprehensive, not skim; "reproduce" → end-to-end, not partial; "compare" → quantitative against a named baseline, not anecdotal.

**Before any plan, fix, or analysis, state four lines:**
1. **Observation** — exact output, exact quote, exact figure number, exact dataset row.
2. **Path** — which step / section / pipeline / source produced it.
3. **Cause statement** — one sentence: "X happens because Y at <location>, which causes W."
4. **Evidence pointer** — file:line, page:section, table/figure ID that confirms the cause, not the symptom.

**Stakeholder vocabulary.** When advisor, reviewer, or collaborator uses different terminology for the same concept, adopt theirs in correspondence with them; do not silently translate.

### Rule 2 — Simplicity First
Minimum code that solves the problem. No speculative features, one-use abstractions, unrequested configurability, or impossible-scenario error handling.

### Rule 3 — Surgical Changes
Touch only what the task requires. Do not improve or reformat adjacent code. Match surrounding style exactly. Remove only imports/variables/functions that **your** change made unused — leave pre-existing dead code untouched.

A single commit (or single review pass) should resolve one focused concern. Mixing paper edits with experiment script changes with bibliography updates makes the change unreviewable and the history unreadable.

### Rule 4 — Goal-Driven Execution
Convert vague instructions into falsifiable goals before starting. For multi-step tasks, publish a numbered plan with per-step verification criteria and confirm before executing. Do not proceed past a step until its verification passes.

### Rule 5 — Pre-Commit Systematic Inspection
Before every commit, audit every touched file across six dimensions:

| Dimension | Check |
|-----------|-------|
| **Contract truth** | No `pass`, `raise NotImplementedError`, or stub bodies. |
| **Orphan config** | Every parameter/config field/env var is consumed downstream. |
| **Orphan return values** | Every non-`None` return is consumed by the caller. |
| **Subsystem connectivity** | No broken wiring, missing DI, or unattached components. |
| **Driver–result alignment** | Every decision-driving field produces an observable effect. |
| **Error visibility** | No silent `except: pass` — every catch re-raises, logs, or converts to typed failure. |

Fix defects before committing. No "I'll fix it later."

## Evidence Integrity

Every claim — in a paper, a literature note, an experiment log, a slide — must be traceable to evidence whose **provenance** is declared.

| Provenance | Meaning | Allowed in |
|---|---|---|
| `measured` | Generated by an actual run / actual experiment / actual reading of the cited source. | All artifacts. |
| `derived` | Computed from `measured` evidence by a documented transformation. | All artifacts; cite the transformation. |
| `simulated` | Generated by a model whose response shape mirrors reality but is not itself measurement. | Internal reasoning only; never headline claims. |
| `illustrative` | Hand-constructed example for explanation. | Tutorial/teaching only; never an empirical claim. |
| `unverified` | Provenance unknown or not yet checked. | Quarantine — treat as `illustrative` until verified. |

**Forbidden:**
- Headline claims backed by `simulated`, `illustrative`, or `unverified` evidence.
- Citations to papers not actually read at the depth the claim requires (no skim-cites).
- Numbers transcribed from a figure without referencing the original table or computation.
- "I'll verify it later." If provenance is `unverified`, the claim is `unverified` until it isn't.

**Test discipline (carried over).** Using mocks to bypass real failures is strictly forbidden. Integration and journey-level tests use real components. A passing test that uses a mock to conceal missing wiring is not evidence. Legitimate mock uses: (1) external network calls in unit tests; (2) fault injection; (3) performance-benchmark stand-ins — document the reason.

## Research Output Maturity Ladder

Every research output declares one of five maturity levels. "Done" without a level is not a valid status.

| Level | Name | Criterion |
|---|---|---|
| M0 | drafted | Written; no internal review; no external eyes. |
| M1 | internally-reviewed | Co-author / advisor / lab-mate read and signed off. |
| M2 | preprint-grade | Methods reproducible from the document alone; figures/tables cite `measured` evidence; ready for arXiv / SSRN / OSF posting. |
| M3 | peer-reviewed | Accepted at a venue with external review; revisions integrated. |
| M4 | replicated | An independent party (not original authors) reproduced the central result. |

Claims of `accepted` / `published` / `done` require M3 minimum. Lower levels report as `IN PROGRESS (level: <Mn>)`.

**Three-part defect closure.** When a defect is found in an artifact (analytic error, miscitation, ambiguous figure, irreproducible step), closure requires: (a) fix in the artifact; (b) a check that prevents recurrence (checklist line, citation-verifier rule, figure-caption template, pre-publication audit row); (c) the process change recorded in this CLAUDE.md or the relevant skill file. Missing any one part = defect remains OPEN.

## Pre-Publication Audit

Before any output moves from M1 → M2 (preprint-grade) or higher, audit across these dimensions. An open finding in any **publication-blocking** dimension blocks promotion — listing it as a "limitation" does not authorize shipping.

| Dimension | Check | Blocking? |
|---|---|---|
| **Citation truth** | Every cited paper has been read at the depth the claim requires. No skim-cites. | yes |
| **Provenance honesty** | Every reported number has a declared `measured` / `derived` provenance. | yes |
| **Reproducibility** | Methods + data + code suffice for a reader to recompute the headline numbers. | yes |
| **Limitations vs. defects** | A genuine scope boundary is a limitation; a known error is a defect and must be fixed before promotion, not relabelled a limitation. | yes |
| **Figure–text parity** | Every figure caption matches the text's claim about it; no orphan panels or unreferenced tables. | advisory |
| **Notation consistency** | Each symbol introduced once; no redefinition mid-paper. | advisory |

## Single Authoritative Document

When N documents reference the same fact (a result, a metric, a citation, a dataset version, a scope statement), exactly one is canonical and the others derive from it.

- For an experiment's headline metric — the experiment log is canonical; paper, slides, and notes derive.
- For a citation — the bibliography file (`.bib`) is canonical; the paper derives.
- For a research direction's scope — the research-direction note is canonical; downstream plans derive.

Restating a canonical fact in a derived document silently is forbidden — link, transclude, or cite the canonical source.

**Released-artifact immutability.** A submitted preprint, accepted paper, or DOI'd dataset is append-only. Corrections become errata or v2 with an explicit diff against v1, never silent overwrites.

## Research Direction Intake

Before adopting a new research direction or topic into the active program, three gates:

- **G1 — Scope gate.** Is this within the lab's / project's stated research program? Off-scope → decline, with a redirect to the right venue.
- **G2 — Redundancy gate.** Is this already answered by published work or an existing internal note? If yes, write the synthesis instead of starting a new line of work.
- **G3 — Verification plan gate.** Is there a falsifiable success criterion stated up-front (which experiment / which dataset / which comparison would settle it)? Without one, the direction stays at exploratory and cannot consume major time.

## Repository Purpose

This is the **Research Intelligence Application (RIA) workspace**. It is *both* a documentation/skills repository (academic research workflows) *and* a software project (the RIA Python package above the hi-agent platform). As of 2026-05-07 (post hi-agent W35-corrective acceptance), RIA's first explicit engineering wave **R-W1** is in flight; sub-wave 1 lands the `real_agent_server` integration fixture, four CI gates, and doc-truth synchronisation. The wave brings v2 architecture (`docs/ria/ria-architecture-v2.md`) to ~95% on-disk coverage. Build via `pip install -e ".[test,lint]"`.

## Structure

- `ria/` — **Research Intelligence Application Python package** (the application above hi-agent platform). Subpackages: `config/`, `platform_client/`, `domain/`, `observability/`, `api/cli/`. Phase 2 adds `orchestration/`, `user/`, `api/{http,mcp}/`. Phase 3 adds `global_layer/`.
- `tests/` — `unit/`, `integration/`, `conformance/` tests for `ria/`.
- `scripts/` — RIA-side CI gate scripts (`check_layering.py`, `check_no_platform_types.py`, `check_idempotency_keys.py`, ...) and operational tooling.
- `skills/` — Research skill definitions (SKILL.md files), each covering a research workflow with steps, templates, and Python tool sketches. Loaded into a PI Agent at runtime via `ria/orchestration/compiler.py` (Phase 2).
- `tools/` — Placeholder for research tools.
- `docs/` — Inter-team communication documents (audit reports, defect logs, platform feedback between hi-agent and RIA teams) **and** RIA's canonical specs under `docs/ria/`.
- `pyproject.toml` — Python project manifest (`ria` package, dev/test extras).
- `ARCHITECTURE.md` — Top-level RIA architecture summary (brief; full canonical at `docs/ria/ria-architecture-v1.md`).

## Four Core Research Skills

Each skill lives in `skills/<name>/SKILL.md` and follows a frontmatter + structured markdown format:

| Skill | Description |
|---|---|
| `literature-review` | Systematic search, evaluation, and synthesis of academic literature |
| `paper-reading` | Structured reading using IMRaD, standardized note templates |
| `paper-writing` | IMRaD-structured academic writing, quality checklists |
| `experiments` | Experiment design, execution monitoring, data quality control |

The `skills/RESEARCH-SKILLS-FRAMEWORK.md` file is the master overview of all four skills and includes full Python tool implementations as reference code.

## Skill File Format

Skill files use this frontmatter format:

```markdown
---
name: "skill-name"
description: "One-line description of when to use this skill."
---
```

Skills include: core capabilities, step-by-step workflows, Python tool class sketches (not implemented — illustrative interfaces only), structured templates (Markdown), best practices, and FAQ.

**Precedence.** When a `skills/*/SKILL.md` specifies a stricter standard for its workflow (e.g. citation depth, figure standards, IMRaD ordering), the skill file wins for that workflow. CLAUDE.md states the principles; skill files state the per-workflow procedures.

## Docs Conventions

Files in `docs/` follow the naming pattern `<source>-to-<target>-<type>-<date>.md` or `<team>-<type>-<date>.md`. These are inter-team audit/defect/feedback documents and are read-only reference material — do not modify them unless explicitly asked.

`docs/ria/` is the canonical spec set for the RIA software side and is editable by the RIA team (versioned via `vN` filenames).

## RIA Posture System

Independent of platform posture but coupled at startup. See `ria/config/posture.py` and `docs/ria/ria-quality-requirements-v1.md` §4.

| RIA `RIA_POSTURE` | Min platform posture | Behaviour |
|---|---|---|
| `dev` | any | permissive; user_id auto-defaulted; budget warnings logged |
| `research` | `research` or `prod` | required user_id, required project_id on writes, required `Idempotency-Key`, per-user budget enforced |
| `prod` | `prod` | research + JWT validation + audit log |

`prod` × platform `dev` exits the RIA process at startup with `exit code 78` (config error).
